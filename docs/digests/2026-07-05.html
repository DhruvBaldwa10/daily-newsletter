Models Smarter, Tools Dumber

Claude Opus 4.8 is hallucinating tool schemas, a $149 open-source release shows what agentic dev actually looks like in practice, and new research suggests scaling alone won't fix social simulation fidelity. The gap between model capability and reliable tool use is today's defining tension.

The Schema Invention Problem: Better Models Breaking Their Own Tools

A strange regression is appearing at the frontier. Armin Ronacher (creator of Flask, Jinja2, and the Pi editor) ran into a specific failure mode with Claude Opus 4.8: the model would call Pi's edit tool with invented fields in the edits[] array — fields that don't exist in the schema. The edit itself was usually correct. The structure was wrong. Not Haiku. Not Sonnet. Opus.

Simon Willison surfaced the post with a pointed summary: newer, more capable models are sometimes worse at strict tool compliance than their predecessors. The theory is that more capable models are more likely to reason about what the tool should accept and generate accordingly, rather than what it actually accepts. Over-generalization at the schema boundary.

This is a real production problem. Any system that validates tool call arguments before execution will start rejecting calls from your most expensive model. Defensive parsing — accepting and stripping unknown fields — helps, but it introduces its own surface area: if the model invents a field called force_overwrite that your parser silently drops, you've lost the signal that the model misunderstood the interface.

The pattern also suggests something about how these models are trained. If RLHF rewards task completion over schema fidelity, and if task completion is easier to verify than argument correctness, models will drift toward inventing plausible-looking arguments. Stricter schema validation in evals — not just output correctness — is the lever here.

"The short version is that newer Claude models sometimes call Pi's edit tool with extra, invented fields in the nested edits[] array. And not Haiku or some small model: Opus 4.8. The edit itself is usually correct but the arguments do not match the schema as the model invents them."

— Armin Ronacher, via Simon Willison's blog

What $149 and Claude Fable Actually Shipped

Simon Willison just published a detailed account of getting sqlite-utils 4.0rc2 out the door with Claude Fable as primary collaborator, total API spend: $149.25. The framing matters — Willison isn't reporting a curiosity, he's documenting a working method for maintaining a serious open-source library (sqlite-utils has ~5k GitHub stars and underpins his datasette ecosystem).

The context: Anthropic is rotating Claude Fable off Max subscriptions within days, which created a forcing function. Willison wanted to reach 4.0 stable before the window closed. He started with a structured prompt laying out the full release checklist, then worked through breaking changes, deprecation notices, documentation, and test coverage in session. The result was rc2, not stable — but he describes the output as something he "felt truly comfortable about," which is a high bar for a SemVer major release.

A few things stand out from the write-up. First, the cost is real and bounded — $149 for a non-trivial engineering session is cheap compared to contractor time, but it's also not zero, and it's per-session, not per-month. Second, the constraint of Fable's imminent unavailability is itself interesting: model transitions are forcing project decisions. Third, this is collaborative maintenance, not code generation — Willison is reviewing, directing, and making judgment calls throughout. The model handles the mechanical surface area; the human holds the design intent.

The broader signal: AI-assisted open-source maintenance is past proof-of-concept. The question now is what the workflow looks like at scale — across more contributors, less-structured codebases, and models that change underneath you.

"Since we only have Claude Fable on our Max subscriptions for a few more days, I decided to see if it could help me get to a 4.0 stable release that I felt truly comfortable about, since I try to keep to SemVer and like my incompatible major versions to be as rare as possible."

— Simon Willison, simonwillison.net

Scaling Won't Fix Social Simulation — And That's a Research Bet Worth Making

A new paper from Stanford and UMass Amherst asks a pointed question: will scaling LLMs close the fidelity gap in social simulation? The authors — Caleb Ziems and colleagues — use scaling laws to study the relationship between compute scale, general capability benchmarks, and simulation fidelity. Their finding: simulation fidelity is largely orthogonal to general capability. Bigger models score better on MMLU and friends, but they don't necessarily produce more faithful simulations of human social behavior.

This matters for anyone building agents that model people — recommendation systems, synthetic user testing, social science tools, customer behavior modeling. The implicit assumption in much of this work is that more capable models are better proxies for human respondents. The paper pushes back hard on that assumption and argues simulation fidelity deserves its own research track, not just better base models.

Separately, the DecompRL paper from Juliette Decugis et al. addresses a different scaling failure: what happens when the base policy has near-zero probability of generating a correct solution? Neither repeated sampling nor RL with verifiable rewards can overcome a search space that's simply too large. Their answer is modular code generation through learned decomposition — break the problem into subproblems the model can solve, then compose. It's a principled argument for why task decomposition isn't just a prompt engineering trick; it's a structural fix for the limits of test-time compute.

"Simulation fidelity is orthogonal to general capabilities and therefore deserving of more research attention."

— Ziems et al., "Will Scaling Improve Social Simulation with LLMs?" (arXiv 2607.02464)

Agentic Search Comes to NASA's Earth Science Archive

A team including researchers from NASA has deployed a public agentic search system for Earth observation datasets — natural language queries over NASA's archive of geoscience tools including Worldview, Giovanni, Harmony, and the Science Discovery Engine. The paper's central claim: knowledge graphs' latent value becomes substantial in the LLM era, when an agent can traverse the KG dynamically rather than requiring users to understand its structure.

This is a narrow domain, but the architecture generalizes. The hard problem in scientific data discovery isn't retrieval — it's that the user often doesn't know the right vocabulary to search with. Domain-specific tools have idiosyncratic names, coverage conventions, and access patterns that experts learn over years. An agentic layer that can map natural language intent to the right combination of NASA tools is genuinely useful, and the public deployment (not just a demo) means there's real usage data coming.

A parallel paper on guard rail validation for AI agents in autonomous telecom networks tackles a related problem: how do you intercept and validate individual agent inference outputs before they trigger live network state changes? Their Guard Rail Validation framework proposes a runtime architecture that classifies decisions by criticality and applies different validation thresholds accordingly. "Autonomy Level 4-5" telecom networks are the stated target, but the framework pattern — criticality-weighted validation gates — maps cleanly to any agentic system touching production state.

"In the era of large language models, the latent value of knowledge graphs can be substantially unlocked — not by replacing KGs with LLMs, but by using LLMs to traverse and query them dynamically."

— Yu et al., "Bringing Agentic Search to Earth Observation Data Discovery" (arXiv 2607.02387)

Cerebras + Hugging Face Put Gemma 4 Into Real-Time Voice

Hugging Face and Cerebras announced they're bringing Gemma 4 to real-time voice AI together — Cerebras' wafer-scale inference hardware paired with Google's Gemma 4 model, accessed through Hugging Face's infrastructure. The collaboration is notable for what it signals about where inference speed matters most right now: voice. Latency tolerances for voice interfaces are an order of magnitude tighter than text generation; users notice 200ms of lag in conversation in a way they don't notice the same delay before a code completion appears.

Cerebras has been the go-to name for raw tokens-per-second benchmarks — their CS-3 systems have posted numbers that make H100 clusters look slow on certain inference workloads. Pairing that with Gemma 4 (which Google released with strong multimodal and instruction-following characteristics) and distributing through Hugging Face's developer ecosystem is a deliberate play to become the default stack for latency-sensitive voice applications.

The timing is interesting. Most voice AI products today use purpose-built speech models (Whisper for ASR, ElevenLabs for TTS, a separate LLM in the middle). The integration bet here is that a single capable model running fast enough can replace that pipeline, reducing latency introduced by multiple model handoffs and simplifying the product architecture. Whether Gemma 4 is actually good enough at the voice-specific dimensions — prosody awareness, interruption handling, spoken language quirks — is the open question the deployment data will answer.

"Real-time voice AI demands sub-100ms response latency — a constraint that fundamentally changes which hardware and model combinations are viable."

— Hugging Face Blog, Cerebras x Gemma 4 announcement

What This All Means

  1. If you're building tool-using agents on frontier models, add schema validation to your eval harness now — not just output correctness. Opus 4.8 inventing fields that don't exist is a production bug, and the fix requires catching it before deployment, not debugging it in logs.
  2. The Willison sqlite-utils session ($149, one rc release) is a useful calibration point for AI-assisted maintenance work: meaningful progress is achievable in a single bounded session, but model availability windows are forcing project timelines in ways that weren't true six months ago. Plan for model transitions in your dev process, not just your product.
  3. If your product uses LLMs to simulate user behavior — for testing, personalization, or social modeling — the Stanford scaling paper is a direct challenge to your assumptions. Fidelity doesn't improve automatically with a better base model; you may need simulation-specific fine-tuning or evaluation to know whether your synthetic users are actually representative.
  4. The Cerebras + Gemma 4 voice integration is worth watching as a template: fast inference hardware plus a capable general model, distributed through a developer platform, aimed at eliminating the multi-model pipeline. If latency is your bottleneck in voice or any real-time feature, this stack is now worth a prototype.
