Sandboxes, Slop, and Reliable Agents

Anthropic publishes a rare deep-dive on how Claude is actually contained across products. A new IBM/Artificial Analysis benchmark catches frontier models below 50% on enterprise IT tasks. And NVIDIA drops an open physical-AI model that could reshape robotics pipelines.

Anthropic Opens the Sandbox Lid

Anthropic published a detailed engineering post — "How we contain Claude across products" — and it's the kind of documentation that almost never gets written. The post breaks down sandbox techniques across Claude.ai, Claude Code, and the API tier, covering process isolation, capability restrictions, and how those controls differ depending on the deployment surface. Simon Willison flagged it immediately, calling it "fantastic" and pointing out a chronic frustration: sandboxing products rarely document how they actually work.

"A complaint I often have about sandboxing products is that they are rarely thoroughly documented, and in the absence of detailed documentation it's hard to know how much I can trust them."

— Simon Willison

That gap matters enormously for anyone deploying Claude in a production environment where the blast radius of an escaped agent or a leaked credential is real. The Anthropic post apparently names specific mechanisms — not just "we use containers" — which makes it genuinely useful for security reviews and compliance conversations.

The timing isn't coincidental. Claude Code is shipping in enterprises now, running long agentic loops with file system and shell access. As the two community HN projects from today illustrate — claude-code-os, an open-source self-updating memory layer for Claude Code, and a Karpathy LLM Wiki-style Obsidian agent workflow — builders are already layering persistent state and operational memory on top of Claude Code. That changes the trust surface considerably. Clear sandbox documentation is table stakes for enterprise adoption; Anthropic publishing it publicly also forces competitors to raise their own documentation bar.

Frontier Models Can't Do Enterprise IT — A New Benchmark Says So

IBM Research and Artificial Analysis released ITBench-AA, the first benchmark focused specifically on agentic enterprise IT tasks — think incident response, configuration management, and operations workflows that real SRE and DevOps teams run. The result is ugly: frontier models score below 50% across the board. Not below human expert level. Below 50% in absolute terms.

This lands differently than typical benchmark news. ITBench-AA isn't measuring trivia or coding puzzles; it's measuring whether an agent can actually close out a ticket in a simulated enterprise environment — the exact use case that a dozen well-funded startups are selling today. The below-50% number across frontier models (which would include GPT-4-class and Claude-class systems) is a hard data point against the "agents are handling Tier-1 IT" pitch.

"Frontier Models Score Below 50% on the First Benchmark for Agentic Enterprise IT Tasks."

— ITBench-AA paper title, Hugging Face Blog / IBM Research

The benchmark connects directly to a companion ArXiv paper from earlier this week, "Counterfactual Evaluation Reveals Hidden Capability Profiles in Clinical LLMs and Agents", which introduces the Causal Sensitivity Score — a metric that probes whether AI systems actually respond to changing inputs or just pattern-match to a fixed output. Together these two papers are making the same argument from different angles: coverage scores flatter agents, and task-grounded evals expose the gap. The AI Snake Oil team made a related point in February with their capability-reliability gap paper. The evidence is now thick enough that "sub-50% on enterprise IT" should be in every enterprise sales deck — on the honest slide.

NVIDIA's Cosmos 3 Is a Physical-AI Bet on Open Models

NVIDIA released Cosmos 3 on Hugging Face today — billed as the first open omni-model designed specifically for physical AI reasoning and action. The model is aimed at robotics and embodied AI pipelines: it reasons about the physical world, not just text or images, and it ships as an open weight model.

The significance here is architectural and strategic simultaneously. On the architecture side, Cosmos 3 is positioned as a world model — one that can reason about states, actions, and consequences in physical environments rather than just generating plausible-sounding text. That puts it in the same conceptual territory as Google DeepMind's robotics work and the embodied AI push at Tesla, but on an open release timeline that anyone can build on today.

On the strategy side, NVIDIA releasing this openly on Hugging Face is a deliberate move. NVIDIA's hardware business benefits directly from every lab and robotics team that builds on Cosmos 3 — more training runs, more inference at the edge, more Jetson deployments. Open-sourcing the model is effectively a demand-generation play for GPUs and the CUDA stack. The timing follows their Blackwell ramp; if physical-AI workloads become the next growth wave, NVIDIA wants the reference architecture to run on their silicon.

"Welcome NVIDIA Cosmos 3: The First Open Omni-model for Physical AI Reasoning and Action."

— Hugging Face Blog, June 1, 2026

For product teams outside robotics, the near-term relevance is in simulation and synthetic data pipelines. World models that understand physical causality produce higher-quality training environments for downstream tasks. Watch for Cosmos 3 to start appearing in fine-tuning and data-augmentation workflows well before anyone ships a physical robot that runs it end-to-end.

Agent Memory Gets a Provenance Architecture

A new ArXiv paper, Eywa: Provenance-Grounded Long-Term Memory for AI Agents, proposes something that anyone who's debugged a RAG pipeline will recognize immediately: the failure modes in agent memory are hard to diagnose because everything — source documents, extracted facts, retrieved context, and the answer model — collapses into a single opaque prompt path. You get a wrong answer, and you can't tell if the evidence was never there, the extraction failed, the retrieval missed it, or the model just ignored it.

Eywa's approach is to enforce a provenance chain — evidence before belief — so that each layer of the memory system can be audited and updated independently. Wrong answer? You can trace it to exactly which step broke down. Stale fact? You can erase it without corrupting the rest of the memory graph.

"A wrong answer may come from missing evidence, unsupported extraction, stale state, retrieval loss, or answer-model behavior."

— Eywa paper abstract, ArXiv 2605.30771

This is directly relevant to anyone building agents that persist across sessions — which is almost everyone building serious agentic products right now. The community projects that surfaced on HN today (claude-code-os's self-updating operational memory, the Obsidian vault-operator workflow) are both attempting ad-hoc versions of exactly this problem. Eywa gives it a formal frame: treat memory as a knowledge graph with explicit provenance, not a blob of text that gets stuffed into context. The practical question is how quickly this translates into something like a LangGraph memory module or an MCP-compatible memory server — the architecture is clean enough that it should.

On the reinforcement learning side, ReuseRL published this week proposes grounding agentic RL in Minimum Description Length — agents generalize better when their successful trajectories compress into reusable skill primitives rather than task-specific shortcuts. It's early research, but the MDL framing for agent generalization is worth watching as a way to think about why current agents are so brittle outside their training distribution.

Evals Are a Skill Gap, Not a Tooling Gap

Hamel Husain dropped evals-skills, an open-source set of skills for AI product evals drawn from working with 50+ companies and teaching 4,000+ students. The framing is pointed: most teams have the wrong problem model. They think they need a better eval tool; what they actually need is to stop making predictable process mistakes before any tool helps.

"Most AI teams focus on the wrong things."

— Hamel Husain, "A Field Guide to Rapidly Improving AI Products"

The companion post from January, "LLM Evals: Everything You Need to Know" (co-authored with Shreya Shankar), distills the most common questions from 700+ engineers and PMs who went through their evals course. The FAQ is explicit that these are "sharp opinions, not universal truths" — which is the right epistemology for a field this young. Two consistent patterns emerge from both pieces: teams reach for LLM-as-judge too fast (before establishing ground truth), and they build eval datasets that don't represent the actual failure distribution in production.

The ITBench-AA result this week is a good example of what rigorous evals look like when they're done with the right questions — task-grounded, with realistic enterprise scenarios, measuring actual completion not plausibility. The contrast with most internal eval suites is stark. Husain's eval-tools post from October makes the same point from the tooling direction: the question isn't "which framework" but "do you know what failure you're trying to catch." Given that evals are now the primary mechanism by which teams decide whether to ship, keep, or roll back agentic features, this is organizational infrastructure, not a side project.

What This All Means

  1. Anthropic's sandbox documentation post is the right artifact to put in front of your security team right now — not as a claim that everything is safe, but as a basis for a real threat-model conversation. If you're deploying Claude Code with file or shell access, build your own containment review against what they've documented.
  2. The ITBench-AA sub-50% result should recalibrate any enterprise AI pitch that promises autonomous IT operations. Benchmark your specific workflows before committing to production — the gap between demo and live incident is where the 50% lives.
  3. Eywa's provenance-grounded memory architecture is the right mental model for any persistent agent you're building today. Whether you adopt the paper's specific design or not, separating evidence, extraction, retrieval, and answer layers makes failure modes diagnosable — and diagnosable failures are fixable ones.
  4. NVIDIA open-sourcing Cosmos 3 as a physical-AI world model is a leading indicator of where inference demand shifts next. If your product touches simulation, synthetic data, or robotics, start evaluating it now; open physical-world models at this capability tier didn't exist six months ago.
