Agents Under Attack, Inside Out

AI agent security moves from theoretical to empirically documented — with numbers that should change how you build. Plus: Google ships managed agents into Gemini API production, a new paper detects doomed agent runs from the first round, and the open agent frontier gets its clearest capability threshold yet.

The Agent Authorization Crisis Has Actual Attack Data Now

Three papers dropped in the last 48 hours that, together, constitute the most rigorous empirical case yet that agent security is not a niche concern — it's a production bug you're shipping today.

The most striking is aiAuthZ, which tested 15 contemporary LLMs against eight real-world attack scenarios drawn from a published corpus of actual agent incidents. Refusal rates ranged from 100% down to 38% — and the most expensive model on the list refused only half the attacks, despite a twentyfold price spread over the cheapest. That last detail lands hard: paying more is not buying you safety.

The paper proposes an off-host authorization gateway that binds tool call permissions to verified identity rather than trusting text in the context window. It's a clean architectural idea, and it addresses the root problem the other papers are also circling: agents act on text they cannot verify, so whoever controls any slice of context can fake authority.

A companion paper, Agent Data Injection Attacks, introduces a new attack category beyond the familiar instruction injection — "data injection," where attacker-controlled content manipulates agent behavior not by impersonating commands but by poisoning the data the agent reasons over. And DualView extends the Dual LLM defense pattern to handle a gap the original missed: when agents persist untrusted data to disk or memory, prior defenses lost track of it. DualView follows tainted data across saves and loads.

"AI agents issue tool calls on the basis of text they cannot verify, so any party who controls part of the context can forge the appearance of authority."

— aiAuthZ paper abstract

Fortress, the stealth Chromium browser for agents announced on HN today, is the infrastructure side of the same problem — keeping agents unblocked while they browse. The security surface that creates is exactly what these papers are mapping.

You Can Tell a Run Will Fail Before It Gets Going

One of the most operationally useful papers in recent memory: "Doomed from the Start" shows that LLM agent failures are predictable from internal representations as early as the first interaction round — using lightweight probes on hidden activations, not expensive behavioral monitors.

The problem the paper addresses is real and expensive: multi-step agent runs that commit to failing trajectories early but keep consuming inference compute until failure becomes observable many steps later. The probes the authors built detect eventual failure from round one, while behavioral scorers watching only outputs were barely better than random at that stage.

The implications for agent product design are practical. Early-abort logic — kill a run when the probe signals doom, restart with a different approach or surface to the user — could cut inference costs on failed runs substantially. More importantly, it reframes agent observability: the signal you want is inside the model's hidden states, not in its outputs.

"Failure is predictable early from the agent's internal representations: lightweight per-round probes on hidden activations anticipate eventual episode failure as early as the first interaction round, where scorers reading only the agent's observable behavior are barely better."

— Doomed from the Start, Ruan et al., arxiv 2607.06503

This connects to a broader trend in agent infrastructure: the monitoring layer is moving from output-level evaluation (did the agent produce the right result?) toward trajectory-level and representation-level signals. SelfMem, another paper from this week's batch, proposes self-optimizing memory that adjusts storage and retrieval strategies based on task feedback — a different kind of internal adaptation, but the same instinct that static evaluation is insufficient for long-horizon agents.

Google Ships Managed Agents to Production — With Background Tasks and Remote MCP

Google quietly pushed a significant feature bundle into the Gemini API yesterday: managed agents now support background task execution, remote MCP server connections, and an expanded set of built-in tools. This isn't a research preview — it's a production API update.

The background task capability is the headline. Managed agents in Gemini can now run asynchronously without holding an open connection, which removes one of the core constraints that pushed developers toward custom orchestration frameworks when they needed long-running work. Paired with remote MCP support, you can now point a Gemini-managed agent at external tool servers without proxying everything through your own backend.

The timing matters. Anthropic's Claude has been the default choice for serious agent builders partly because of Claude's strong instruction-following in agentic contexts, but Google is closing the infrastructure gap quickly. Gemini's managed agent layer is now handling concerns — session management, tool routing, background scheduling — that most teams were building themselves six months ago.

"Managed agents feature bundle launch" — background tasks, remote MCP and more now available in Gemini API.

— Google AI Blog, Mariano Cocirio, July 7 2026

For product teams deciding where to build long-running agent infrastructure, the choice is less obvious than it was at the start of Q2. Google's managed layer trades some control for operational simplicity; the question is whether the abstraction holds at the edge cases your users will find. The remote MCP support in particular deserves evaluation — it's now a first-class primitive rather than a workaround, which changes what's reasonable to promise in an agent product roadmap.

GLM-5.2 and the Open Agent Capability Threshold

Nathan Lambert called it a "step change" — and he's been careful with that phrase. GLM-5.2, released last month by Tsinghua's KEG lab, crossed what Lambert describes as a capability threshold he's been monitoring for open-weight models in agentic tasks. Not just better benchmark numbers, but qualitatively different behavior on the kinds of multi-step, tool-using tasks that closed-source models like GPT-4 and Claude have dominated.

The distinction Lambert draws between open and closed models is important context: he's argued separately that open and closed models are currently on different exponential curves, with open models lagging on the tasks where marginal intelligence compounds. GLM-5.2 may mark where those curves start to converge for agent-specific workloads.

"A capability threshold I've been carefully monitoring."

— Nathan Lambert, Interconnects, on GLM-5.2

For builders, an open-weight model crossing a meaningful agent capability threshold changes the cost calculus on a specific class of products: anything requiring customization, self-hosting for data residency, or fine-tuning on proprietary task data. GLM-5.2 isn't a drop-in replacement for Claude Sonnet 5 on general tasks — but if Lambert's read is right, it's the first open model where deploying it as an agent backbone on specific domains isn't a significant capability concession. That's a different kind of competitive moat than raw benchmark performance.

Formal Verification Meets Code Agents — and the New Benchmark for SQL-Native AI

Two benchmark papers from this week redefine what "code generation" means when the target is not just correct code but provably correct code, and when the SQL being generated calls LLM functions directly.

The formal verification paper, "Harnessing Code Agents for Automatic Software Verification," tackles Coq proofs — the hardest end of the correctness spectrum. Previous LLM approaches to proof generation wired a fixed, human-designed proof strategy into the system, then constrained the model to follow it. The new work lets agents develop their own proof strategies, which is a meaningful architectural shift: the agent is doing search and planning over proof space rather than executing a predetermined template. The authors show this generalizes better across theorem types than prior constrained approaches.

The SQL side is just as interesting. Spider 2.0-AIFunc benchmarks 465 verified instances of "AI-native SQL" — queries that call LLM capabilities (classification, sentiment analysis, similarity search) as native SQL functions within cloud platforms like BigQuery and Snowflake. Existing text-to-SQL benchmarks give you no signal on whether models can generate these hybrid queries. The benchmark covers 125 real-world databases.

"Major cloud data platforms now expose large language model capabilities as native SQL functions, enabling analysts to perform classification, filtering, sentiment analysis, extraction, similarity search, and aggregation within ordinary SQL queries. Yet existing text-to-SQL benchmarks evaluate only conventional SQL."

— Spider 2.0-AIFunc, Liu et al., arxiv 2607.06229

Spider 2.0-AIFunc matters because AI-native SQL is already shipping in production — BigQuery ML, Snowflake Cortex, and Redshift ML all expose LLM functions via SQL — and the benchmark finally creates a way to measure whether your text-to-SQL pipeline handles them. If you're building on top of any of those platforms, this is the eval gap you should be closing.

What This All Means

  1. Agent authorization needs to be an architecture decision, not a prompt decision. The aiAuthZ data — a twentyfold price spread buying no meaningful safety improvement — means your current "we use a good model" defense is not a defense. Evaluate off-host authorization gateways that bind tool calls to verified identity before your agent product ships anything consequential.
  2. Build early-abort into any long-running agent loop now. The "Doomed from the Start" finding is production-ready in spirit even if the probes need tuning for your stack — instrument round-one behavior, define abort criteria, and stop burning compute on trajectories your own system can predict will fail.
  3. Google's managed agent API update means the build-vs-buy calculus on agent infrastructure shifted yesterday. Background tasks and remote MCP as first-class primitives are exactly what teams were rolling themselves; audit whether your custom orchestration is now redundant before you maintain it another quarter.
  4. Spider 2.0-AIFunc is the eval you need if you're building on BigQuery, Snowflake, or Redshift. AI-native SQL — queries that call LLM functions inline — is already in production at those platforms, and there's now a 465-instance benchmark to measure whether your generation pipeline handles it. Run your current pipeline against it before claiming text-to-SQL coverage.
