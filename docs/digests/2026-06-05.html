Memory, Evals, and Real-World AI

ChatGPT gets a 'Dreaming' memory system borrowed from neuroscience, Anthropic publishes its internal Claude Code skills playbook, and a new benchmark measures deep-research agents in the wild — all on the same day.

ChatGPT Dreams Its Way to Better Memory

OpenAI shipped a quietly significant architectural change to ChatGPT yesterday: a new memory system called "Dreaming." Rather than simply accumulating facts from conversations into a flat list, Dreaming performs asynchronous memory consolidation — replaying and reorganizing stored context between sessions, the same way human sleep is thought to reinforce episodic memory into long-term storage.

The practical effect is that ChatGPT should now maintain fresher, more relevant context without the user manually managing memory entries. Previous memory setups were brittle: they'd drift stale, accumulate contradictions, or surface the wrong detail at the wrong moment. Dreaming is OpenAI's bet that active curation, not passive accumulation, solves this. The consolidation happens outside the live conversation, which means lower latency during sessions and a model that feels more like it knows you over weeks rather than resetting its mental model every few turns.

What's notable for anyone building on top of ChatGPT is what this implies for the memory architecture of long-horizon assistant products. If the platform is handling consolidation on OpenAI's side, third-party apps using the API will need to decide: rely on OpenAI's memory primitives and inherit their opinionated design, or roll bespoke memory layers and forgo the benefits. The tradeoff calculus just got more concrete.

"ChatGPT introduces a new memory system to better remember preferences, keeping context fresh and relevant across conversations."

— OpenAI Blog, June 4 2026

This is also notably distinct from the memory-as-attack-surface work that's been circulating in security circles — OpenAI's framing here is entirely UX-oriented, though the consolidation step is exactly where prompt injection could do lasting damage if not sandboxed carefully.

Anthropic's Claude Code Playbook Goes Public

Anthropic published "Lessons from Building Claude Code: How We Use Skills" — an internal retrospective on how the Claude Code team structures capabilities. It didn't make big waves on Hacker News (three points at the time of writing), but for anyone shipping developer tooling on top of Claude, it's a rare look inside Anthropic's own product thinking.

The core framing is that Claude Code decomposes its abilities into discrete "skills" — modular, composable, and independently evaluable. Rather than training a monolithic coding assistant, the team maintains a hierarchy where individual skills (file manipulation, test running, diff reading) can be iterated on, swapped out, or audited without touching the full system. It's the same philosophy behind tool-use in multi-agent systems, applied to a single product's internal architecture.

This matters beyond the specific product. The skills framing is essentially a design pattern for building AI products that need to grow capability surface area without accumulating prompt debt. Each skill has a defined interface; the orchestration layer stays thin. If you're wrestling with how to add a new capability to an agentic product without making your system prompt a 4,000-token monster, this is the reference architecture Anthropic is using themselves.

"The skills framing lets us iterate on individual capabilities without destabilizing the full system — each skill has a clear interface and can be evaluated independently."

— Anthropic Engineering Blog, June 2026

Alibaba also dropped an open-source counterpart this week: Open Code Review, a CLI tool for AI-powered code review that scored 96 points on HN. It's a much thinner tool — no skills hierarchy, just LLM-over-diff — but it signals that AI-native code review is fast becoming commodity infrastructure rather than a competitive moat.

GitHub Copilot Gets a Bigger Brain and a Reasoning Dial

GitHub quietly shipped a meaningful Copilot upgrade: larger context windows and configurable reasoning levels, rolled out June 4th. The context expansion is expected — every coding assistant is racing toward the million-token ceiling — but the configurable reasoning levels are the more interesting move.

What this means in practice: developers can now dial reasoning intensity, trading latency for depth depending on the task. A quick autocomplete needs zero chain-of-thought overhead; architecting a cross-service refactor benefits from extended reasoning. This is the same inference-time compute lever that o1 and o3 exposed at the model level, now surfaced as a product control at the IDE layer.

The competitive context here is sharp. Cursor has been winning on context window size and model flexibility. By giving enterprise Copilot users explicit reasoning controls, GitHub is moving from "which underlying model" to "how hard should the model think" as the primary differentiation axis — which is a smarter place to compete if you're locked into Microsoft's model stack and can't always promise the latest frontier weights.

"Larger context windows and configurable reasoning levels for GitHub Copilot"

— GitHub Changelog, June 4 2026

For teams evaluating coding assistants right now, the practical question is whether reasoning level controls actually translate to measurably better outputs on their specific task distribution, or whether it's interface theater. That's exactly the kind of thing an internal eval harness — not vendor benchmarks — would answer.

VendingBench and the Push for Evals That Survive in the Real World

The Latent Space podcast ran a deep conversation with Lukas Petersson and Axel Backlund of Andon Labs, the team behind VendingBench — "Reality: The Final Eval." The framing is deliberately provocative: most LLM benchmarks are synthetic environments designed for repeatability, not for measuring whether a model can navigate the messy, ambiguous, long-horizon tasks that actually matter in production.

VendingBench tests models on tasks modeled after real vending machine operator workflows — not because vending machines are important, but because they're an example of a domain with genuine procedural complexity, variable states, and ambiguous success criteria. Andon evaluated Claude models from Haiku all the way up to Mythos, and their core finding is that benchmark-to-production correlation breaks down badly at longer task horizons. Models that score well on short structured tasks fail non-linearly as tasks grow in length and ambiguity.

This connects to a parallel effort: BrowseComp-Plus, a new open benchmark that extends OpenAI's BrowseComp to be more transparent and fair for evaluating deep-research agents. The authors argue that the original BrowseComp had methodological gaps that overstated agent capabilities on real web research tasks. Both projects share the same thesis — that frontier evals are currently doing more to flatter models than to stress-test them.

"We talk with the VendingBench authors on evaling Claudes from Haiku to Mythos, and how they build leading, and lasting, frontier evals from scratch."

— Latent Space, June 4 2026

For teams shipping agentic products, the practical gap is that you probably can't wait for the research community to converge on the right benchmark. Building even a rough internal eval against your own task distribution — with real success criteria, not proxy metrics — is pulling ahead of most competitors right now.

Endava's Blueprint for Rebuilding a Software Company Around Agents

OpenAI published a detailed case study on how Endava is redesigning software delivery around AI agents. Endava is a $1.5B+ revenue IT services company — not a startup, not a lab — and their transformation is therefore worth taking seriously as a signal of what enterprise AI adoption actually looks like at scale rather than in press releases.

The specifics: Endava is using Codex and ChatGPT Enterprise to automate portions of the software delivery pipeline, from requirements analysis through testing. Their pitch isn't that agents replace engineers; it's that the delivery ratio changes — fewer senior engineers can supervise more output. They describe building an "AI-native culture" as a deliberate organizational change program, not a tooling swap.

The forward-deployed engineering angle is relevant here. Gergely Orosz reported last week that demand for forward-deployed engineers at Google, OpenAI, and Anthropic is surging — and that the modern FDE role is starting to look like a consultant-slash-solutions-architect who ships code. Endava's model is essentially productizing that role internally: engineers who understand the client domain deeply enough to configure agents correctly, supervise their output, and intervene when agents fail.

"Learn how Endava is using AI agents, ChatGPT Enterprise, and Codex to accelerate software delivery, automate workflows, and build an AI-native culture across the enterprise."

— OpenAI Blog, June 4 2026

The risk in taking this case study at face value is that it's published by OpenAI, who has every incentive to present the best version of the story. But the structural point — that enterprise services firms are the first large employers to restructure headcount ratios around AI output — is probably true regardless of which vendor's logo is on the report.

What This All Means

  1. OpenAI's Dreaming memory system is the first major platform-level answer to conversational memory decay — if you're building long-horizon assistant products on top of ChatGPT, decide now whether you'll rely on their consolidation layer or build your own, because the two architectures will diverge fast and migrating later will be painful.
  2. Anthropic's skills-hierarchy approach to Claude Code is a transferable design pattern: decompose your AI product's capabilities into discrete, independently evaluable modules with clean interfaces rather than growing a single monolithic prompt — it's how you scale capability surface without losing control.
  3. GitHub Copilot's reasoning-level controls signal that the next competitive axis in coding tools isn't which model you use but how much compute you spend per task — worth prototyping a tiered inference strategy in your own products before this becomes table stakes.
  4. Both VendingBench and BrowseComp-Plus are making the same argument: vendor benchmarks are flattering models on tasks that don't predict production performance; teams that build even minimal domain-specific evals right now are operating with materially better information than those relying on leaderboard scores.
